为什么需要理解原型

MOTIVATION

🤔 class 不是"真正的"继承

昨天我们学了 class 和 extends，但 JavaScript 其实没有传统意义的"类"。所有面向对象的能力，都是靠原型（Prototype）实现的。

class 只是语法糖——它让写法更优雅，但底层仍然是原型机制。不理解原型，会遇到这些坑：

  • 🔴 继承时引用类型属性被"共享"（多个实例互相影响）
  • 🔴 this 在原型方法中丢失指向
  • 🔴 无法判断属性是"自己的"还是"继承的"
  • 🔴 看不懂 CesiumJS 源码中的原型操作
ℹ️
关键理念：JavaScript 是基于原型的语言（Prototype-Based Language）。每个对象都有一个隐藏的"原型对象"，方法实际上是从原型对象"借"来的。这和 Java/C++ 的"基于类的语言（Class-Based）"是根本性的区别。
2

原型对象：每个函数都有 prototype

PROTOTYPE

📁 函数自带一个 prototype 属性

在 JavaScript 中，每个函数（包括普通函数、构造函数、箭头函数例外）都会自动拥有一个 prototype 属性，它指向一个对象——这个对象就是"原型对象"。

// 任何普通函数都有 prototype
function sayHello() {}
console.log(typeof sayHello.prototype);  // "object"
console.log(sayHello.prototype);            // { constructor: sayHello }

// 构造函数也一样
function Marker(name, lat, lon) {
  this.name = name;
  this.lat  = lat;
  this.lon  = lon;
}

// 往 prototype 上添加方法
Marker.prototype.describe = function() {
  return `${this.name} 位于 (${this.lat}, ${this.lon})`;
};

// 创建实例
const m1 = new Marker('北京', 39.9, 116.4);
const m2 = new Marker('上海', 31.2, 121.5);

// 实例上并没有 describe 方法，但能调用（从原型上找到的）
console.log(m1.describe());  // "北京 位于 (39.9, 116.4)"
console.log(m2.describe());  // "上海 位于 (31.2, 121.5)"
console.log(m1.describe === m2.describe);  // true（共享同一个函数）
💡
共享方法的威力：把方法放在 prototype 上，所有实例共享同一个函数引用，节省内存。如果把方法写在构造函数里，每个实例都会创建一份独立的函数副本。
3

__proto__ 与 Object.getPrototypeOf

LINK

🔗 两条线：prototype 与 __proto__

很多人搞混这两个属性。理解它们的关系是掌握原型的关键：

function Marker(name) { this.name = name; }
Marker.prototype.describe = function() {
  return this.name;
};

const m = new Marker('北京');

// 1. 函数（类）这一边：prototype 属性
console.log(Marker.prototype);     // { describe: [Function], constructor: Marker }

// 2. 实例这一边：__proto__ 属性（指向构造函数的 prototype）
console.log(m.__proto__);           // { describe: [Function], constructor: Marker }

// 3. 它们是同一个对象！
console.log(m.__proto__ === Marker.prototype);  // true

// 4. 推荐的现代写法（__proto__ 已不推荐使用）
console.log(Object.getPrototypeOf(m) === Marker.prototype);  // true

📊 图示：prototype 与 __proto__ 的关系

    ┌─────────────────────┐
    │   Marker (函数)      │
    │                     │
    │   prototype  ───────┼──────┐
    └─────────────────────┘      │ 指向
                                 ▼
    ┌─────────────────────┐    ┌─────────────────────┐
    │  m (实例对象)        │    │  prototype 对象      │
    │                     │    │                     │
    │  name: '北京'        │    │  describe: [Function]│
    │  __proto__  ────────┼───▶│  constructor: Marker│
    └─────────────────────┘    └─────────────────────┘

简单记忆：函数有 prototype（指向原型对象），实例有 __proto__（指向构造函数的 prototype）。两者指向同一个对象。

⚠️
不要在生产代码中使用 __proto__！它是浏览器厂商私自加的特性，虽然 ES6 标准化了，但性能较差且语义模糊。永远用 Object.getPrototypeOf(obj) 和 Object.setPrototypeOf(obj, proto) 替代。
4

原型链：属性查找的路径

CHAIN

🔎 查找规则：自己没有，就找原型

当访问一个对象的属性时，JavaScript 引擎会按以下顺序查找：

  1. 先在对象自身查找，找到就用
  2. 找不到就去 __proto__（即构造函数的 prototype）查找
  3. 还找不到就继续沿着 __proto__ 向上找
  4. 直到 Object.prototype，再往上就是 null
  5. 到了 null 还没找到，返回 undefined

这条链式结构就是原型链（Prototype Chain）。它是 JavaScript 实现继承的核心机制。

// 任何对象的原型链终点都是 Object.prototype
const obj = { a: 1 };

console.log(obj.a);        // 1（自己的属性）
console.log(obj.toString); // [Function] ← Object.prototype 上的方法！

// 验证原型链
console.log(obj.__proto__ === Object.prototype);  // true
console.log(Object.prototype.__proto__);   // null ← 原型链终点

// 数组的原型链
const arr = [1, 2, 3];
console.log(arr.map);    // [Function] ← Array.prototype 上的方法
console.log(arr.toString); // [Function] ← 继承自 Object.prototype

📊 完整原型链图示

    m (Marker实例)
    │
    │  __proto__
    ▼
    Marker.prototype      { describe, constructor: Marker }
    │
    │  __proto__
    ▼
    Object.prototype      { toString, hasOwnProperty, ... }
    │
    │  __proto__
    ▼
    null                  ← 链的终点
ℹ️
性能提示：原型链越长，属性查找越慢。理想情况下不超过 3 层。Vue、React 内部都刻意保持原型链扁平化以优化性能。
5

方法的"复用"机制

REUSE

🔁 两种方法定义方式的区别

// ========== 方式1：方法定义在构造函数里（不推荐）==========
function MarkerV1(name) {
  this.name = name;
  this.describe = function() {  // 每个实例都创建一份！
    return this.name;
  };
}

// ========== 方式2：方法定义在 prototype 上（推荐）==========
function MarkerV2(name) {
  this.name = name;
}
MarkerV2.prototype.describe = function() {  // 所有实例共享一份
  return this.name;
};

// 验证差异
const a1 = new MarkerV1('A');
const a2 = new MarkerV1('A');
console.log(a1.describe === a2.describe);  // false（两份独立函数）

const b1 = new MarkerV2('B');
const b2 = new MarkerV2('B');
console.log(b1.describe === b2.describe);  // true（同一份函数）

⚠️ 经典坑：原型上的引用类型

function Team() {}
Team.prototype.members = [];   // 默认空数组

const t1 = new Team();
const t2 = new Team();

t1.members.push('Alice');
console.log(t2.members);  // ['Alice'] ← 被影响了！

// 原因：t1 和 t2 共用同一个 members 数组（原型上的引用）

这就是为什么构造函数里写 this.xxx = []（实例属性）比 prototype.xxx = [] 更安全——每个实例拥有独立的副本。

6

六种继承方式对比

PATTERNS

📚 历史上的 6 种继承方案

在 ES6 class 出现之前，前辈们用各种"土办法"实现继承。每种都有缺陷，了解它们能让你更深入理解原型。

方式优点缺点
原型链继承 简单 引用类型属性共享；无法传参
构造函数继承 可传参；属性独立 方法无法复用（每个实例都创建）
组合继承 兼具两者优点 调用两次父构造函数
原型式继承 适合对象间的继承 引用类型共享（同原型链）
寄生式继承 增强对象 方法无法复用
寄生组合继承 最完美！ 实现稍复杂

📝 方式 1：原型链继承

function Parent() {
  this.name = 'parent';
  this.colors = ['red', 'blue'];
}
Parent.prototype.say = function() {
  console.log(this.name);
};

function Child() {}
Child.prototype = new Parent();  // ← 关键：子类的 prototype 指向父类实例

const c1 = new Child();
const c2 = new Child();
c1.colors.push('green');
console.log(c2.colors);  // ['red', 'blue', 'green'] ← 共享了！

📝 方式 2：构造函数继承（call）

function Parent(name) {
  this.name = name;
  this.colors = ['red', 'blue'];
}
Parent.prototype.say = function() {
  console.log(this.name);
};

function Child(name, age) {
  Parent.call(this, name);  // ← 关键：借调父构造函数
  this.age = age;
}

const c1 = new Child('Alice', 20);
const c2 = new Child('Bob', 25);
c1.colors.push('green');
console.log(c2.colors);  // ['red', 'blue'] ← 互不影响了！
// c1.say(); ← 报错！say 是父类 prototype 上的，c1 找不到

寄生组合继承（最优解）

BEST PRACTICE

🏆 业界公认的完美继承模式

寄生组合继承结合了"借用构造函数"（属性独立）和"原型链"（方法共享）的优点，只调用一次父构造函数，没有冗余属性。这是 ES6 class 内部采用的方案。

// 工具函数：创建"干净的"子类原型对象
function inheritPrototype(Child, Parent) {
  // 1. 创建父类原型的副本（浅拷贝）
  const prototype = Object.create(Object.getPrototypeOf(Parent.prototype));
  // 2. 修复 constructor 指向
  prototype.constructor = Child;
  // 3. 把新原型赋给子类
  Child.prototype = prototype;
}

// ===== 父类 =====
function MapEntity(name) {
  this.name = name;
  this.tags = [];
}
MapEntity.prototype.describe = function() {
  return `${this.name} (tags: ${this.tags.join(', ')})`;
};

// ===== 子类 =====
function Marker(name, lat, lon) {
  MapEntity.call(this, name);  // 继承属性
  this.lat = lat;
  this.lon = lon;
}

inheritPrototype(Marker, MapEntity);  // 继承方法

// 添加子类自己的方法
Marker.prototype.moveTo = function(lat, lon) {
  this.lat = lat;
  this.lon = lon;
};

// ===== 测试 =====
const m1 = new Marker('北京', 39.9, 116.4);
const m2 = new Marker('上海', 31.2, 121.5);

m1.tags.push('首都');
console.log(m1.describe());  // "北京 (tags: 首都)"
console.log(m2.describe());  // "上海 (tags: )" ← 互不影响！

console.log(m1 instanceof Marker);     // true
console.log(m1 instanceof MapEntity);  // true
console.log(m1.moveTo(40.0, 116.5));
console.log(m1.lat, m1.lon);  // 40 116.5
ℹ️
记住这个模式：用 Object.create() 创建"干净"的父类原型副本，避免调用两次父构造函数。这是理解 ES6 class extends 实现原理的关键。

class 语法背后的原型真相

DESUGAR

🍴 class ≈ 寄生组合继承 + 语法糖

ES6 的 class 实际上就是用原型和构造函数实现的"伪装"。下面这段 class 代码：

// ===== 你写的 class 代码 =====
class MapEntity {
  constructor(name) {
    this.name = name;
  }
  describe() {
    return this.name;
  }
}

class Marker extends MapEntity {
  constructor(name, lat, lon) {
    super(name);
    this.lat = lat;
    this.lon = lon;
  }
  moveTo(lat, lon) {
    this.lat = lat;
    this.lon = lon;
  }
}

被 JavaScript 引擎翻译成的等价代码（核心部分）：

// ===== 引擎内部做的事 =====
function MapEntity(name) {
  this.name = name;
}
MapEntity.prototype.describe = function() {
  return this.name;
};

function Marker(name, lat, lon) {
  MapEntity.call(this, name);   // ← super(name)
  this.lat = lat;
  this.lon = lon;
}

// ← extends 的本质：Object.setPrototypeOf 串联原型链
Object.setPrototypeOf(Marker.prototype, MapEntity.prototype);

// 修复 constructor 指向
Marker.prototype.constructor = Marker;

// 添加子类方法
Marker.prototype.moveTo = function(lat, lon) {
  this.lat = lat;
  this.lon = lon;
};
💡
验证方法：打开浏览器控制台，输入 Marker.prototype.__proto__ === MapEntity.prototype，会返回 true。这证实了 class extends 的本质就是原型链操作。

对象操作：hasOwnProperty / in / instanceof

INSPECTION

🔍 区分"自己的"与"继承的"

原型链让属性"无处不在"，但有时候你需要知道一个属性到底来自哪里：

方法检查范围返回
obj.hasOwnProperty('key') 只看自身 boolean
'key' in obj 自身 + 原型链 boolean
obj instanceof Class 原型链上是否有 Class.prototype boolean
Object.getPrototypeOf(obj) 取原型对象 对象/null 
function Marker(name) {
  this.name = name;        // 自己的属性
}
Marker.prototype.describe = function() { return this.name; };

const m = new Marker('北京');

// hasOwnProperty：只看自己
console.log(m.hasOwnProperty('name'));     // true
console.log(m.hasOwnProperty('describe')); // false（在原型上）
console.log(m.hasOwnProperty('toString')); // false（继承自 Object）

// in 操作符：包括原型链
console.log('name'     in m);  // true
console.log('describe' in m);  // true ← 通过原型链能找到
console.log('toString' in m);  // true
console.log('xxx'      in m);  // false

// instanceof：检查原型链
console.log(m instanceof Marker);      // true
console.log(m instanceof Object);      // true（所有对象最终继承自 Object）
console.log(m instanceof Array);       // false

// 遍历自身属性（最常用）
const ownKeys = Object.keys(m);            // ['name']
const allKeys = for(const k in m);          // name, describe, ...
console.log("ownKeys:", ownKeys);
⚠️
hasOwnProperty 的陷阱：如果对象用 Object.create(null) 创建（没有原型），它就没有 hasOwnProperty 方法。安全做法：Object.prototype.hasOwnProperty.call(obj, 'key')。

🌎 CesiumJS 关联场景

CESIUM

🌐 场景 1：CesiumJS 内部的类继承体系

CesiumJS 大量使用原型继承。打开 Cesium 源码，你会看到这样的代码（简化版）：

// Cesium 内部常见的继承模式
function defineProperties(Class, properties) {
  const subDefines = [];  // 收集子类的 getter/setter
  for (const name in properties) {
    // 用 Object.defineProperty 给原型定义属性
    Object.defineProperty(Class.prototype, name, properties[name]);
  }
  // ...
}

// 父类 Cartesian3
function Cartesian3(x, y, z) {
  this.x = x;
  this.y = y;
  this.z = z;
}

// 给原型加方法
Cartesian3.prototype.magnitude = function() {
  return Math.sqrt(this.x ** 2 + this.y ** 2 + this.z ** 2);
};

// 创建实例：c.__proto__ === Cartesian3.prototype
const c = new Cartesian3(1, 2, 2);
console.log(c.magnitude());  // 3（共享原型方法）

  }
}

🌐 场景 3：自定义 Entity 类型的继承体系

实际 CesiumJS 项目中，常用类继承管理各种地理实体：

// 父类：基础地理实体
class GeoEntity {
  constructor(options) {
    this.id = options.id || crypto.randomUUID();
    this.name = options.name || '未命名实体';
    this.createdAt = Date.now();
    this.metadata = options.metadata || {};
  }
  getAge() {
    return Date.now() - this.createdAt;
  }
}

// 子类：地理标记
class GeoMarker extends GeoEntity {
  constructor(options) {
    super(options);  // ← 父类构造函数（实际是 GeoEntity.call(this, options)）
    this.lat = options.lat;
    this.lon = options.lon;
    this.altitude = options.altitude || 0;
  }
  toCartesian() {
    // 实际 CesiumJS: return Cesium.Cartesian3.fromDegrees(this.lon, this.lat, this.altitude);
    return { x: this.lon, y: this.lat, z: this.altitude };
  }
}

// 子类的子类：带图标的标记
class IconMarker extends GeoMarker {
  constructor(options) {
    super(options);
    this.icon = options.icon || '📍';
    this.color = options.color || '#61dafb';
  }
  render() {
    return `${this.icon} ${this.name}`;
  }
}

// 使用
const m = new IconMarker({
  name: '天安门', lat: 39.9054, lon: 116.3976, icon: '🏛️'
});
console.log(m.render());     // "🏛️ 天安门"
console.log(m.getAge());      // 创建至今的毫秒数
console.log(m.toCartesian()); // {x: 116.3976, y: 39.9054, z: 0}

🌐 场景 4：判断对象类型 — instanceof 的实战

在 CesiumJS 中处理各种实体时，经常需要判断类型：

// CesiumJS 的 Entity 有多种类型
function handleEntityClick(entity) {
  // instanceof 适用于用 class 创建的实例
  if (entity instanceof Cesium.Entity) {
    console.log('点击了一个 Entity');
  }

  // hasOwnProperty 检查特定字段（更通用）
  if (entity.position && entity.hasOwnProperty('billboard')) {
    console.log('这是一个图标标注');
  }

  // in 操作符检查原型链上的方法（适合 Cesium 这种用原型扩展的库）
  if ('flyTo' in entity) {
    console.log('该对象支持 flyTo 方法');
  }

  // 实际项目中的典型模式：组合判断
  const type = entity.billboard ? '图标'
            : entity.polyline ? '线'
            : entity.polygon ? '面'
            : '其他';
  console.log(`实体类型：${type}`);
}
💡
原型知识在 CesiumJS 中的应用：CesiumJS 源码几乎全部用原型机制组织（早期代码用 function 构造函数 + prototype，新代码用 class）。理解原型链能让你阅读 CesiumJS 源码时不被绕晕，也能写出更高效的扩展代码。

📋 今日知识点总结

  • ✅ 每个函数都自带 prototype 属性，指向一个原型对象
  • ✅ 每个实例都有 __proto__，指向其构造函数的 prototype
  • ✅ 推荐用 Object.getPrototypeOf 代替 __proto__
  • ✅ 原型链是属性查找的路径：自己 → 父类 → Object.prototype → null
  • ✅ 方法放在 prototype 上共享，属性放在 this 上独立
  • ✅ 6 种继承方式中，寄生组合继承是 ES6 class 内部采用的方案
  • ✅ class 本质是语法糖，引擎内部用原型实现继承
  • ✅ hasOwnProperty 只看自己，in 包含原型链
  • ✅ 理解原型是阅读 CesiumJS 源码、扩展内置类、调试继承问题的基础
