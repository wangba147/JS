同步 vs 异步：为什么需要异步？

CONCEPT

JavaScript 是单线程语言，同一时刻只能执行一个任务。如果所有操作都"同步"执行，遇到网络请求、文件读取等耗时操作时，页面会完全卡死。

🔄 同步执行

代码按顺序一行一行执行，前一行没完成，后一行必须等待。遇到耗时操作时整个程序被"阻塞"。

// 同步：每一步都必须等待上一步完成
console.log("任务 1 开始");
console.log("任务 2 开始");  // 必须等 1 完成
console.log("任务 3 开始");  // 必须等 2 完成

// 假设有一个耗时同步操作
function heavySyncTask() {
  const start = Date.now();
  while (Date.now() - start < 3000) {} // 卡 3 秒
  return "完成";
}
console.log("开始");
const result = heavySyncTask();  // 阻塞 3 秒
console.log("结果:", result);

⚡ 异步执行

遇到耗时操作时，不等待结果，继续执行后续代码。等耗时操作完成后，再回头处理结果（通过回调、Promise 等机制）。

// 异步：耗时操作"放后台"，主线程继续工作
console.log("1. 开始");

// setTimeout 是异步的典型例子
setTimeout(() => {
  console.log("3. 2秒后执行");
}, 2000);

console.log("2. 立即执行（不等待 setTimeout）");

// 输出顺序：1 → 2 → (2秒后) 3
💡
核心思想：异步让程序"在等待时做别的事"。网络请求、定时器、用户交互、文件读取都是典型的异步场景。CesiumJS 加载瓦片数据、相机飞行动画都依赖异步机制。
特性 同步（Synchronous） 异步（Asynchronous）
执行顺序 严格顺序，阻塞 非阻塞，结果"晚到"
耗时操作 卡住整个程序 放后台，主线程继续
用户体验 页面冻结、卡顿 流畅、响应快
典型应用 数学计算、循环 网络请求、定时器、事件

事件循环：JS 的"调度员"

EVENT LOOP

JavaScript 通过事件循环（Event Loop）机制实现异步。它是一个不断循环的"调度员"，负责把异步任务的回调放到合适时机执行。

🏗️ 运行机制：调用栈 + 任务队列

理解事件循环需要知道两个关键结构：

  • 调用栈（Call Stack）：当前正在执行的函数调用，LIFO（后进先出）
  • 任务队列（Task Queue）：等待执行的异步回调，FIFO（先进先出）
  • 微任务队列（Microtask Queue）：优先级更高的任务队列（Promise 回调）
/* 事件循环的执行顺序规则：
   1. 执行调用栈中的同步代码
   2. 同步代码执行完毕后，先清空微任务队列
   3. 从宏任务队列中取一个任务执行
   4. 重复 2、3
*/

console.log("1. 同步");

setTimeout(() => {
  console.log("3. setTimeout (宏任务)");
}, 0);

Promise.resolve().then(() => {
  console.log("2. Promise (微任务)");
});

console.log("4. 同步结束");

// 输出顺序：1 → 4 → 2 → 3
// 关键：微任务比宏任务优先执行！
ℹ️
宏任务 vs 微任务：setTimeout、setInterval、I/O 是宏任务；Promise.then、queueMicrotask、MutationObserver 是微任务。每一轮事件循环中，微任务队列会全部清空，然后才执行一个宏任务。

🎯 经典面试题：执行顺序

console.log("A");

setTimeout(() => console.log("B"), 0);

new Promise((resolve) => {
  console.log("C");  // 同步执行
  resolve();
}).then(() => console.log("D"));

console.log("E");

// 答案：A → C → E → D → B
// 解析：同步代码 A、C、E 先执行
//       然后微任务 D
//       最后宏任务 B

事件绑定与 Event 对象

EVENTS

用户点击、鼠标移动、键盘按键、滚动页面……这些事件（Event）都是网页交互的基础。JavaScript 通过事件监听器（Event Listener）来响应它们。

📌 三种事件绑定方式

① HTML 内联绑定（不推荐）

<!-- HTML 中直接写 onclick -->
<button onclick="handleClick()">点击我</button>

② DOM 属性绑定（早期方案）

const btn = document.getElementById("myBtn");
btn.onclick = function() {
  console.log("被点击了！");
};
// 缺点：只能绑定一个函数，后绑定的会覆盖前面的

③ addEventListener（推荐）

const btn = document.getElementById("myBtn");

// 可以绑定多个监听器，按顺序执行
btn.addEventListener("click", () => {
  console.log("第一个监听器");
});

btn.addEventListener("click", () => {
  console.log("第二个监听器");
});

// 移除监听器（需要先保存引用）
const handler = () => console.log("一次性");
btn.addEventListener("click", handler);
btn.removeEventListener("click", handler);

🎁 Event 对象：事件的"身份证"

当事件触发时，浏览器会创建一个Event 对象，传递给监听器函数，包含事件的所有信息：

document.addEventListener("click", (event) => {
  console.log("事件类型:", event.type);           // "click"
  console.log("触发元素:", event.target.tagName); // 被点击的标签
  console.log("鼠标坐标:", event.clientX, event.clientY);
  console.log("是否按下Ctrl:", event.ctrlKey);
  console.log("时间戳:", event.timeStamp);
});

// 常用属性：
// event.target      - 实际触发事件的元素
// event.currentTarget - 绑定监听器的元素
// event.preventDefault() - 阻止默认行为（如表单提交）
// event.stopPropagation() - 停止事件传播

📋 常用事件类型

事件流：捕获与冒泡

EVENT FLOW

当点击一个嵌套元素时，事件其实在 DOM 树中"走"了两遍：

🌊 三个阶段

  1. 捕获阶段（Capturing）：事件从 window 一路向下传到目标元素
  2. 目标阶段（Target）：事件到达目标元素
  3. 冒泡阶段（Bubbling）：事件从目标元素向上冒泡到 window
<!-- DOM 结构 -->
<div id="outer">      <!-- 外层 -->
  <div id="inner">    <!-- 内层 -->
    <button id="btn">点击</button>
  </div>
</div>

/* 点击按钮时，事件传播路径：
   window → document → html → body → outer → inner → btn (捕获)
   btn (目标)
   btn → inner → outer → body → html → document → window (冒泡)
*/

🎯 第三个参数：useCapture

// addEventListener(type, listener, useCapture)
// useCapture: false（默认）= 冒泡阶段触发
// useCapture: true = 捕获阶段触发

const outer = document.getElementById("outer");
const inner = document.getElementById("inner");
const btn = document.getElementById("btn");

// 捕获阶段（从上到下）
outer.addEventListener("click", () => console.log("1. outer 捕获"), true);
inner.addEventListener("click", () => console.log("2. inner 捕获"), true);

// 冒泡阶段（从下到上）
inner.addEventListener("click", () => console.log("3. inner 冒泡"));
outer.addEventListener("click", () => console.log("4. outer 冒泡"));

// 点击 button 时的输出顺序：1 → 2 → 3 → 4

🚫 停止传播

// 阻止事件继续传播（不再触发上层/下层元素）
btn.addEventListener("click", (e) => {
  e.stopPropagation();
  console.log("事件到此为止");
});

// 阻止默认行为（保留事件传播）
document.querySelector("a").addEventListener("click", (e) => {
  e.preventDefault();  // 阻止链接跳转
});

// 两者同时阻止
e.stopPropagation();
e.preventDefault();

事件委托：高效处理大量元素

DELEGATION

如果给 1000 个列表项都绑定点击事件，性能会很差。事件委托利用事件冒泡，只在父元素上绑定一次监听器，通过 event.target 判断是哪个子元素触发的。

❌ 传统方式：每个元素都绑定

// 假设有 1000 个 li
const items = document.querySelectorAll("li");
items.forEach(item => {
  item.addEventListener("click", () => {
    console.log("点击了：", item.textContent);
  });
});
// 问题：1000 个监听器，内存占用大
// 新增 li 后还要重新绑定

✅ 事件委托：只在父元素绑定一次

// 利用事件冒泡，只在 ul 上绑定一个监听器
const list = document.getElementById("myList");

list.addEventListener("click", (e) => {
  // 检查点击的是不是 li
  if (e.target.tagName === "LI") {
    console.log("点击了：", e.target.textContent);
  }
});

/* 优势：
   1. 性能好 - 只绑定 1 个监听器
   2. 动态生效 - 后续新增的 li 也能响应
   3. 内存省 - 监听器数量少
*/
💡
CesiumJS 中的应用：CesiumJS 的 InfoBox（点击实体弹出的信息框）、图层面板工具栏，都是用事件委托实现的。地球上有上万个 Entity，逐个绑定事件是不可能的。

🎯 更稳健的事件委托（用 data-* 属性）

<!-- HTML -->
<ul id="toolbar">
  <li data-action="zoom-in">放大</li>
  <li data-action="zoom-out">缩小</li>
  <li data-action="reset">复位</li>
</ul>

// JS：用 data-* 属性分发到不同处理器
const actions = {
  "zoom-in": () => console.log("放大地图"),
  "zoom-out": () => console.log("缩小地图"),
  "reset": () => console.log("复位视角")
};

document.getElementById("toolbar").addEventListener("click", (e) => {
  const action = e.target.dataset.action;
  if (action && actions[action]) {
    actions[action]();
  }
});

回调函数与回调地狱

CALLBACK

回调函数（Callback）是异步编程最基本的形式：把一个函数作为参数传递给异步操作，操作完成后调用它。

✅ 简单回调示例

// 异步读取文件，操作完成后调用回调
fetch("https://api.example.com/data")
  .then(response => response.json())
  .then(data => console.log(data));

// 模拟：异步获取数据（用 setTimeout 模拟）
function getUserData(userId, callback) {
  setTimeout(() => {
    const user = { id: userId, name: "张三", age: 28 };
    callback(user);
  }, 1000);
}

getUserData(1, (user) => {
  console.log("用户:", user.name);
});

💀 回调地狱（Callback Hell）

当多个异步操作有依赖关系时（一个的结果是另一个的输入），回调会层层嵌套，形成"金字塔"：

// 经典回调地狱：获取用户 → 获取订单 → 获取商品 → 渲染页面
getUser(userId, (user) => {
  getOrders(user.id, (orders) => {
    getProducts(orders[0].productId, (product) => {
      getReviews(product.id, (reviews) => {
        renderPage(user, orders, product, reviews, () => {
          console.log("终于渲染完了...");
        });
      });
    });
  });
});

/* 问题：
   1. 代码横向膨胀，难以阅读
   2. 错误处理要写多次
   3. 维护噩梦
*/
ℹ️
解决方案：ES6 引入的 Promise 把回调地狱"摊平"成链式调用，ES2017 引入的 async/await 进一步让异步代码看起来像同步代码（Day 11 会深入学习）。

🔄 用 Promise 改善（预览）

// 同样逻辑用 Promise 写
getUser(userId)
  .then(user => getOrders(user.id))
  .then(orders => getProducts(orders[0].productId))
  .then(product => getReviews(product.id))
  .then(reviews => renderPage(reviews))
  .catch(err => console.error(err));
// 链式调用，扁平清晰！

定时器：setTimeout 与 setInterval

TIMERS

定时器是 JavaScript 内置的异步机制，不阻塞主线程。

⏰ setTimeout：延时执行

// setTimeout(回调, 延迟毫秒数, ...参数)
const timerId = setTimeout(() => {
  console.log("2秒后执行");
}, 2000);

// 取消未执行的定时器
clearTimeout(timerId);

// 传额外参数
setTimeout((name, age) => {
  console.log(`Hello ${name}, ${age}`);
}, 1000, "张三", 28);

🔁 setInterval：循环执行

// 每隔 1 秒执行一次
let count = 0;
const intervalId = setInterval(() => {
  count++;
  console.log(`第 ${count} 次`);
  if (count >= 5) {
    clearInterval(intervalId);  // 停止
  }
}, 1000);

// ⚠️ 注意：setInterval 不保证精确间隔
// 如果回调耗时 1.5 秒，间隔会变成 2.5 秒

🎬 requestAnimationFrame：动画专用

专门为动画设计，与浏览器刷新频率同步（通常 60fps = 每 16.67ms 一次）。

let angle = 0;
function animate() {
  angle += 1;
  console.log(`当前角度：${angle}°`);

  // 继续下一帧
  requestAnimationFrame(animate);
}
requestAnimationFrame(animate);

/* 优势：
   - 与显示器刷新率同步，不掉帧
   - 页面隐藏时自动暂停（省电）
   - 适合做 CesiumJS 的相机动画、模型旋转
*/
⚠️
定时器的"坑"：浏览器的最小延迟通常是 4ms（嵌套 5 层以上）；标签页在后台时，setTimeout/setInterval 会被节流到 1 秒一次；this 在回调中可能丢失（用箭头函数解决）。
API用途取消方式
setTimeout延迟执行一次clearTimeout(id)
setInterval周期性执行clearInterval(id)
requestAnimationFrame动画帧cancelAnimationFrame(id)

CesiumJS 关联场景

CESIUMJS

今天学的事件与异步机制，在 CesiumJS 开发中几乎无处不在。下面看几个真实场景：

🌍 场景 1：监听鼠标事件获取地理坐标

用户点击地球时，把屏幕坐标转换为经纬度：

// CesiumJS 经典用法
const viewer = new Cesium.Viewer("cesiumContainer");

// 用事件委托方式监听屏幕点击
const handler = new Cesium.ScreenSpaceEventHandler(viewer.scene.canvas);

handler.setInputAction((click) => {
  // click.position 是屏幕坐标 {x, y}
  const cartesian = viewer.scene.camera.pickEllipsoid(
    click.position,
    viewer.scene.globe.ellipsoid
  );

  if (cartesian) {
    // 把 3D 笛卡尔坐标转成经纬度
    const cartographic = Cesium.Cartographic.fromCartesian(cartesian);
    const lon = Cesium.Math.toDegrees(cartographic.longitude);
    const lat = Cesium.Math.toDegrees(cartographic.latitude);

    console.log(`点击位置: ${lon.toFixed(4)}°E, ${lat.toFixed(4)}°N`);
  }
}, Cesium.ScreenSpaceEventType.LEFT_CLICK);

⏱️ 场景 2：相机飞行动画（异步 + 事件）

调用 flyTo 后，相机在指定时间内飞向目标，可以监听完成事件：

// 异步相机飞行，1 秒后到达北京上空
viewer.camera.flyTo({
  destination: Cesium.Cartesian3.fromDegrees(116.40, 39.90, 50000),
  duration: 1.0,
  complete: () => {              // 飞行完成（异步回调）
    console.log("已到达北京！");
  },
  cancel: () => {                // 被用户取消（异步回调）
    console.log("飞行被取消");
  }
});

// 监听每一帧（用 requestAnimationFrame 思想）
viewer.scene.preRender.addEventListener(() => {
  // 每一帧渲染前触发，60fps 频率
  const height = viewer.camera.positionCartographic.height;
  console.log(`当前高度: ${height.toFixed(0)}m`);
});

📡 场景 3：异步加载 3D Tiles 数据

CesiumJS 加载大规模 3D 模型（如倾斜摄影、点云）都是异步的：

// 异步加载 3D Tiles 瓦片数据
const tileset = await Cesium.Cesium3DTileset.fromUrl(
  "./data/tiles/tileset.json",
  {
    // 加载完成的回调
    onLoaded: (tileset) => {
      console.log("3D 瓦片加载完成");
      viewer.scene.primitives.add(tileset);
    }
  }
);

// 监听加载进度（事件）
tileset.loadProgress.addEventListener((numberOfPendingRequests) => {
  console.log(`还有 ${numberOfPendingRequests} 个瓦片在加载...`);
});

🎯 场景 4：实体点击事件委托

当场景中有多个 Entity（点、线、面、模型）时，监听点击事件：

// CesiumJS 的 ScreenSpaceEventHandler 内部就是事件委托
handler.setInputAction((click) => {
  // pick 找到被点击的 Entity
  const picked = viewer.scene.pick(click.position);

  if (picked && picked.id) {
    // clicked on a Cesium Entity
    const entity = picked.id;
    console.log(`点击了: ${entity.name}`);

    // 根据类型分发处理
    if (entity.point)   showPointInfo(entity);
    if (entity.billboard) showBillboardInfo(entity);
    if (entity.model)   showModelInfo(entity);
  }
}, Cesium.ScreenSpaceEventType.LEFT_CLICK);
💡
学习路线：掌握今天的事件和异步基础后，后续学习 Promise/async/await 时，CesiumJS 的所有"返回 Promise 的 API"（如 Cesium3DTileset.fromUrl、IonResource.fromAssetId）就能用上了。今天的事件绑定是 CesiumJS 交互的根基。

📋 今日知识点总结

  • ✅ 同步代码按顺序阻塞执行；异步代码"不等待"，结果通过回调或 Promise 传递
  • ✅ 事件循环机制：执行栈 → 清空微任务 → 取一个宏任务 → 重复
  • ✅ addEventListener(type, handler, useCapture) 是推荐的事件绑定方式，第三个参数控制捕获或冒泡
  • ✅ 事件流三阶段：捕获 → 目标 → 冒泡；用 stopPropagation 阻止传播
  • ✅ 事件委托利用冒泡，在父元素上统一处理子元素事件，性能更好且支持动态元素
  • ✅ 回调函数是异步基础，但多层嵌套会形成"回调地狱"
  • ✅ setTimeout/setInterval/requestAnimationFrame 三种定时器适用不同场景
  • ✅ CesiumJS 的相机飞行、3D Tiles 加载、实体点击全部依赖事件与异步机制
