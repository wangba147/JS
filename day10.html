为什么需要错误处理

WHY

💥 现实：代码不可能一次就写对

从第一天学 JavaScript 开始，你已经遇到了无数个 Uncaught TypeError、ReferenceError。这不是你菜——是常态：

  • 🔴 用户输入了非法的坐标（lat=91 越界）
  • 🔴 CesiumJS 加载 3D Tiles 失败（网络问题）
  • 🔴 GeoJSON 文件格式错误、字段缺失
  • 🔴 后端 API 返回 500，response.json() 解析失败
  • 🔴 用户浏览器不支持 WebGL，初始化崩溃
⚠️
没有错误处理的代价：用户在地图上看到一个白屏、点不到任何东西、控制台一堆红字却不知道是哪一步出错。好的错误处理 = 用户看到友好提示 + 开发者能看到详细日志。

🎯 本课目标

  • ✅ 认识 JavaScript 的 7 种内置错误类型
  • ✅ 熟练使用 try/catch/finally
  • ✅ 学会主动 throw 异常、定义自己的错误类
  • ✅ 掌握异步代码中的错误处理（Promise、async/await）
  • ✅ 用 window.onerror、unhandledrejection 兜底全局
  • ✅ 掌握 console 调试全家桶和 DevTools 断点技巧

JavaScript 的错误类型

ERRORS

📚 7 个内置错误构造函数

JavaScript 内置了 7 种错误类型，它们都是 Error 的子类。了解这些类型，看错误信息就能知道是哪类问题：

new Error('通用错误');                  // 所有错误的基类
new SyntaxError('语法错误');           // eval 解析失败等
new ReferenceError('引用错误');        // 访问未声明的变量
new TypeError('类型错误');              // 类型不匹配，最常见
new RangeError('范围错误');            // 数组越界、递归爆栈
new URIError('URI 错误');               // encodeURI/decodeURI 失败
new EvalError('eval 错误');             // 几乎遇不到

🔍 实战案例：常见错误触发

// 1. ReferenceError —— 使用未声明的变量
console.log(undeclaredVar);    // Uncaught ReferenceError

// 2. TypeError —— 类型不匹配（最常见）
const pos = null;
pos.x();                       // Uncaught TypeError: Cannot read property 'x' of null
console.log(123().toString);   // 数字不是函数

// 3. RangeError —— 数组越界 / 递归爆栈
const arr = [1, 2];
arr[99] = 3;                  // OK（不报错）
arr.push(3).length;           // TypeError，push 不返回数组

// 4. SyntaxError —— 语法写错（eval 中可捕获）
eval('const x =; ');              // SyntaxError

// 5. URIError —— 编码失败
decodeURI('%');                  // URIError: URI malformed
💡
最常见的 3 种错误：SyntaxError（写错代码）、ReferenceError（拼错变量名）、TypeError（在 null/undefined 上调用方法）。在 CesiumJS 中，TypeError 最为常见，比如对未创建的 Entity 调用方法。

try/catch/finally 基础

TRY-CATCH

🔨 三段式：try / catch / finally

这是 JavaScript 错误处理的基本骨架：

try {
  // 尝试执行的代码（可能抛错）
  const data = JSON.parse(jsonString);
  console.log('解析成功', data);
} catch (err) {
  // 出错时进入这里（err 是 Error 对象）
  console.error('解析失败:', err.message);
} finally {
  // 无论是否出错都会执行（可选）
  console.log('清理工作完成');
}

🎯 实战：解析用户输入的坐标

function parseCoordinate(input) {
  try {
    const coord = JSON.parse(input);

    // 主动校验
    if (typeof coord.lat !== 'number' || typeof coord.lon !== 'number') {
      throw new TypeError('坐标必须是数字');
    }

    if (coord.lat < -90 || coord.lat > 90) {
      throw new RangeError('纬度必须在 -90 到 90 之间');
    }
    if (coord.lon < -180 || coord.lon > 180) {
      throw new RangeError('经度必须在 -180 到 180 之间');
    }

    return coord;
  } catch (err) {
    // 区分错误类型，给用户友好提示
    if (err instanceof SyntaxError) {
      return { error: '输入不是合法 JSON' };
    } else if (err instanceof TypeError) {
      return { error: '坐标类型错误: ' + err.message };
    } else if (err instanceof RangeError) {
      return { error: '坐标越界: ' + err.message };
    } else {
      return { error: '未知错误: ' + err.message };
    }
  }
}

console.log(parseCoordinate('{lat: 39.9, lon: 116.4}'));  // 合法
console.log(parseCoordinate('{"lat": "abc", "lon": 116}')); // TypeError
console.log(parseCoordinate('{"lat": 91, "lon": 0}'));  // RangeError

🔄 控制流：try/catch 不会终止后续代码

// 关键：try/catch "吃掉"了错误，程序不会崩
console.log('1. 开始');

try {
  undefinedFunction();   // ReferenceError，但被捕获
} catch (e) {
  console.log('2. 捕获到错误:', e.name);
}

console.log('3. 继续执行');  // 依然会执行

// 输出：
// 1. 开始
// 2. 捕获到错误: ReferenceError
// 3. 继续执行
💡
finally 的妙用：无论 try/catch 怎么走，finally 都会执行。最适合释放资源：关闭数据库连接、停止 loading 动画、清理临时文件。即使 return 也不会跳过它！
⚠️
try/catch 的代价：V8 引擎在 V8 6.0+ 已经优化了 try/catch 的性能，几乎无开销。但不要把整个文件用 try 包起来，那样会让其他正常错误也被吞掉，难以调试。只在"可能出错"的小段代码块周围使用。

throw：主动抛出异常

THROW

📣 不止 try-catch，你也可以"主动抛错"

当遇到不该继续执行的非法状态时，主动 throw 比让程序继续跑下去更安全：

// 1. 抛字符串（不推荐，看不出类型）
throw '出错了';  // 不推荐

// 2. 抛 Error 对象（推荐）
throw new Error('连接数据库失败');

// 3. 抛特定类型（更精准）
throw new TypeError('参数必须是字符串');

// 4. 抛自定义对象（高级用法）
throw { code: 401, msg: '未授权', retryable: false };

🎯 CesiumJS 风格的参数校验

// CesiumJS 内部大量使用这种风格的参数校验
function flyToCoordinate(longitude, latitude, height) {
  // 防御性检查
  if (typeof longitude !== 'number') {
    throw new TypeError('longitude 必须是数字，实际收到: ' + typeof longitude);
  }
  if (typeof latitude !== 'number') {
    throw new TypeError('latitude 必须是数字');
  }
  if (latitude < -90 || latitude > 90) {
    throw new RangeError('latitude 超出 [-90, 90] 范围: ' + latitude);
  }
  if (longitude < -180 || longitude > 180) {
    throw new RangeError('longitude 超出 [-180, 180] 范围');
  }
  if (height !== undefined && height < 0) {
    throw new RangeError('height 不能为负数');
  }

  // 实际业务...
  return `已飞行到 (经=${longitude}, 纬=${latitude}, 高=${height || 0})`;
}

// 调用
try {
  flyToCoordinate('北京', 39.9);  // 抛 TypeError
} catch (e) {
  console.error(e.message);
}
ℹ️
"抛 vs 返"的原则：如果错误是调用方能立即处理（比如重试、给默认值），用返回值（return { error: ... }）；如果错误是无法恢复的非法状态（比如参数错得离谱），用 throw。

自定义错误类

CUSTOM

🏦 继承 Error 创建业务专属错误

在大型项目（特别是 CesiumJS 这种库开发）中，自定义错误类让你的错误处理更清晰：

// 自定义错误基类
class GeoAppError extends Error {
  constructor(message, code, context) {
    super(message);
    this.name = 'GeoAppError';
    this.code = code;
    this.context = context || {};
    this.timestamp = new Date().toISOString();
  }
}

// 派生出具体的业务错误
class InvalidCoordinateError extends GeoAppError {
  constructor(coord, reason) {
    super(
      `无效坐标 (lat=${coord.lat}, lon=${coord.lon}): ${reason}`,
      'INVALID_COORD',
      { coord, reason }
    );
    this.name = 'InvalidCoordinateError';
  }
}

class TileLoadError extends GeoAppError {
  constructor(url, httpStatus) {
    super(`3D Tiles 加载失败: ${url} (HTTP ${httpStatus})`, 'TILE_LOAD_FAIL', { url });
    this.name = 'TileLoadError';
  }
}

// 使用：精准捕获特定错误
function addMarker(entity) {
  const c = entity.position;
  if (!c || c.lat < -90 || c.lat > 90) {
    throw new InvalidCoordinateError(c || {}, '纬度越界');
  }
  // ...
}

try {
  addMarker({ position: { lat: 999, lon: 100 } });
} catch (e) {
  if (e instanceof InvalidCoordinateError) {
    console.warn('坐标错误，请检查输入。Code:', e.code);
  } else {
    throw e;  // 未知错误继续往上抛
  }
}
💡
自定义错误的好处：instanceof 判断更精准，可以附加业务字段（错误码、上下文、原始数据），让上层能根据错误码做不同处理（如 401 跳登录、500 重试）。

Error 对象的属性

PROPERTIES

📂 3 个核心属性

任何 Error 实例都有这些属性：

try {
  throw new TypeError('参数错');
} catch (err) {
  console.log(err.name);     // "TypeError"（错误类名）
  console.log(err.message);  // "参数错"（人读）
  console.log(err.stack);    // "TypeError: 参数错\n  at ...（调用栈）"
  console.log(err.toString()); // "TypeError: 参数错"
}

🔍 堆栈追踪（Stack Trace）的妙用

err.stack 记录了错误发生的完整调用链——这是调试的"金矿"：

function c() {
  throw new Error('爆炸了');
}
function b() {
  c();
}
function a() {
  b();
}
try {
  a();
} catch (e) {
  console.log(e.stack);
  // 输出：
  // Error: 爆炸了
  //   at c (script.js:2:9)
  //   at b (script.js:5:3)
  //   at a (script.js:8:3)
  //   at <global> (script.js:11:3)
  // 含义：c 抛错，被 b 调用，b 被 a 调用，a 从全局直接调用
}
ℹ️
格式化堆栈：Node.js 提供了 Error.captureStackTrace(target, constructorOpt)，可以隐藏构造函数自身的栈帧。CesiumJS 内部用得很巧妙——包装错误时去掉中间的"胶水代码"层，只暴露给开发者真正业务相关的栈。

异步代码的错误处理

ASYNC

🔁 Promise 链：catch 兜底

// 1. 经典 Promise 链式
fetch('/api/cities')
  .then(res => {
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return res.json();
  })
  .then(data => console.log(data))
  .catch(err => console.error('加载城市失败:', err));

// 2. Promise.all 的批量处理
Promise.all([
  loadTileset('cityA'),
  loadTileset('cityB'),
  loadTileset('cityC')
])
  .then(results => addToScene(results))
  .catch(err => console.error('某个 tileset 加载失败:', err));

🚀 async/await：try/catch 回归

async/await 最大的好处就是可以像同步代码一样用 try/catch：

// async 函数中可以正常用 try/catch
async function loadCityData(cityName) {
  try {
    showLoading(`正在加载 ${cityName}...`);

    const response = await fetch(`/api/cities/${cityName}`);
    if (!response.ok) {
      throw new Error(`城市 ${cityName} 不存在 (HTTP ${response.status})`);
    }
    const data = await response.json();

    // 数据校验
    if (!data.lat || !data.lon) {
      throw new TypeError('城市数据缺少坐标字段');
    }

    return data;
  } catch (err) {
    console.error('加载失败:', err.message);
    // 上报错误到日志服务
    reportError(err);
    // 返回默认值（可选）
    return null;
  } finally {
    hideLoading();  // 无论如何都关掉 loading
  }
}
⚠️
async 函数的"陷阱"：async 函数总是返回 Promise。如果在 async 函数内部 throw 但没被 try/catch 捕获，外层用 .then() 链时需要再加 .catch()。否则错误会变成"unhandled promise rejection"（未处理的 Promise 拒绝）。

🔨 Promise.allSettled：宽容的并发

// 与 Promise.all 不同：allSettled 不会因为一个失败就全盘放弃
const results = await Promise.allSettled([
  loadLayer('buildings'),
  loadLayer('roads'),
  loadLayer('poi')   // 即使这个失败，别的也照常加载
]);

results.forEach(r => {
  if (r.status === 'fulfilled') {
    addLayer(r.value);
  } else {
    console.warn('图层加载失败:', r.reason.message);
  }
});

全局错误捕获

GLOBAL

🏠 三大全局兜底事件

总有一些错误会从 try/catch 缝隙中溜走——浏览器提供 3 个事件"网兜"住它们：

// 1. window.onerror：捕获所有未处理的同步错误
window.addEventListener('error', (event) => {
  console.log('全局错误:', event.message);
  console.log('出错文件:', event.filename);
  console.log('行号:', event.lineno);
  console.log('列号:', event.colno);
  console.log(`  at ${event.filename}:${event.lineno}`);

  // 重要：返回 true 阻止默认上报到控制台
  return true;
});

// 2. unhandledrejection：捕获未处理的 Promise 拒绝
window.addEventListener('unhandledrejection', (event) => {
  console.error('Promise 拒绝:', event.reason);
  // event.promise 是被拒绝的 Promise
  // event.reason 是拒绝原因（通常是 Error 对象）
  // 上报到日志服务：
  reportError(event.reason);
});

// 3. console.error 的全局代理（高级）
const origConsoleError = console.error;
console.error = function(...args) {
  reportError(args.join(' '));  // 上报
  origConsoleError.apply(console, args);  // 继续打印
};
💡
生产环境的标配：把上面 3 个监听注册到应用入口，再把错误上报到日志服务（如 Sentry、Fundebug、自己的后端）。这样即使用户没反馈，开发者也能看到崩溃现场。
⚠️
注意：window.onerror 无法捕获资源加载失败（图片、script 404）和 Promise 内部错误（要用 unhandledrejection）。要全面兜底，需要结合多个事件。

console 调试技巧全家桶

CONSOLE

🎬 7 个常用方法

// 1. console.log —— 最常用，打印值
console.log('北京', { lat: 39.9, lon: 116.4 });

// 2. console.table —— 数组/对象以表格展示（最爱）
console.table([
  { name: '北京', lat: 39.9, lon: 116.4, pop: 2154 },
  { name: '上海', lat: 31.2, lon: 121.5, pop: 2487 },
  { name: '广州', lat: 23.1, lon: 113.3, pop: 1867 }
]);

// 3. console.group / groupEnd —— 分组折叠（避免刷屏）
console.group('城市加载流程');
  console.log('1. 解析 URL');
  console.log('2. 发起请求');
  console.group('3. 响应处理');
    console.log('3.1 解析 JSON');
    console.log('3.2 校验字段');
  console.groupEnd();
console.groupEnd();

// 4. console.time / timeEnd —— 性能计时
console.time('加载 tiles');
await loadTileset('bigCity');
console.timeEnd('加载 tiles');  // "加载 tiles: 1234.5ms"

// 5. console.assert —— 断言（不通过时打印）
console.assert(data.lat >= -90 && data.lat <= 90, '纬度越界!', data);

// 6. console.count —— 计数（看代码执行次数）
function render() {
  console.count('render 调用次数');
  // ...
}
// 渲染 60 次后会看到: render 调用次数: 60

// 7. console.trace —— 打印当前调用栈
console.trace('谁调用了我？');

CesiumJS 关联场景

CESIUMJS

🌐 场景 1：资源加载失败

3D Tiles、地形、影像经常会因为网络或路径问题加载失败：

async function safeLoadTileset(url) {
  try {
    const tileset = await Cesium.Cesium3DTileset.fromUrl(url);
    console.log('tileset 加载成功', url);
    return tileset;
  } catch (err) {
    console.error('tileset 加载失败:', url);
    console.error('原因:', err.message);

    // 给用户友好提示
    showToast('3D 模型加载失败，请检查网络或 URL: ' + url);

    // 降级方案：尝试加载低分辨率版本
    if (!url.includes('_lq/')) {
      const fallbackUrl = url.replace('tiles/', 'tiles/_lq/');
      console.log('尝试降级版本:', fallbackUrl);
      return safeLoadTileset(fallbackUrl);
    }
    return null;
  }
}

🌍 场景 2：WebGL 不可用

用户浏览器可能不支持 WebGL，初始化前要检测：

function initCesium() {
  // 检测 WebGL 支持
  try {
    const canvas = document.createElement('canvas');
    const gl = canvas.getContext('webgl2') || canvas.getContext('webgl');
    if (!gl) {
      throw new Error('浏览器不支持 WebGL');
    }
  } catch (e) {
    showFallbackUI('您的浏览器不支持 3D 地球显示，请使用 Chrome / Edge / Firefox 最新版');
    return;
  }

  // 安全创建 Viewer
  try {
    const viewer = new Cesium.Viewer('cesiumContainer', {
      terrainProvider: await loadTerrain()
    });
    return viewer;
  } catch (e) {
    console.error('Cesium 初始化失败:', e);
    showFallbackUI('3D 地球初始化失败:' + e.message);
  }
}

function validateGeoJSON(data) {
  if (data.type !== 'FeatureCollection') {
    throw new TypeError('不是有效的 FeatureCollection');
  }
  if (!Array.isArray(data.features)) {
    throw new TypeError('features 字段缺失或不是数组');
  }

  // 用 console.warn 报告但不阻断的"软错误"
  data.features.forEach((f, i) => {
    if (!f.geometry) {
      console.warn(`feature[${i}] 缺少 geometry，已跳过`);
    }
  });

  // 用 console.assert 校验关键字段
  console.assert(data.features.length > 0, 'GeoJSON 中没有任何要素');
}

📊 场景 4：错误上报与监控

生产环境必备——把错误上报到日志服务：

// 错误上报服务（模拟）
function reportError(err, extra = {}) {
  const payload = {
    message: err.message,
    stack: err.stack,
    name: err.name,
    url: location.href,
    userAgent: navigator.userAgent,
    timestamp: Date.now(),
    ...extra
  };

  // 用 navigator.sendBeacon 保证页面卸载时也能发出去
  if (navigator.sendBeacon) {
    navigator.sendBeacon('/api/errors', JSON.stringify(payload));
  }
}

// 全局监听
window.addEventListener('error', (e) => reportError(e.error || new Error(e.message)));
window.addEventListener('unhandledrejection', (e) => reportError(e.reason));
💡
CesiumJS 错误处理的"潜规则"：CesiumJS 在出错时通常会调用 throwDeveloperError（开发者错误，会断言）或 throwRuntimeError（运行时错误）。前者说明你用错了 API，后者通常是环境问题。在生产环境，通常把 Cesium.Ion.defaultAccessToken 设好、用 requestRenderMode 减少报错，并用 try/catch 包裹关键调用。
